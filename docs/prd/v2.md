## 1. 版本目标与范围

### 1.1 基线现状（v0.1.0）

v0.1.0 已交付可运行的知识库骨架：

| 能力 | 现状 |
|------|------|
| 记忆溯源 | ✅ 已实现 `provenance` / `trust_level` / `review_status` |
| 混合检索 | ✅ Tantivy 全文 + SQLite BLOB 向量 + RRF 融合 |
| 文档上传 | ✅ Markdown / TXT / PDF（PDF 为 ASCII fallback） |
| 前端 | ✅ React + Vite + Tailwind，支持上传、搜索、审核/图谱占位页 |
| 嵌入 | ⚠️ `RandomEmbedder` 占位，真实 ONNX 实现被注释 |
| 权限 | ⚠️ 仅 header 透传 `X-Space-Id` / `X-User-Id`，无 RBAC |
| 协同编辑 | ❌ 未实现 |
| Agent API / MCP | ❌ 未实现 |
| 部署 | ⚠️ 基础 Docker Compose + 简单 CI，无 Helm/K8s/生产配置管理 |

### 1.2 v0.2.0 目标

**将 epicode-kb 从“可运行骨架”升级为具备真实语义理解、多人协同、企业级权限、Agent 接入与生产部署能力的人机协同企业知识库。**

本版本重点推进并交付以下 5 个方向：

1. **真实嵌入/解析**：替换 `RandomEmbedder`，接入 ONNX 真实嵌入；补全 PDF/DOCX/Markdown/TXT 解析与分块元数据。
2. **协同编辑**：多用户实时协同编辑记忆/文档；版本历史与回退；冲突可视化与解决。
3. **权限**：JWT 认证 + RBAC；知识库/空间/记忆三级权限；公开/团队/私有可见性。
4. **Agent 集成**：Agent 可消费 API/SDK；MCP（Model Context Protocol）接口；Agent 写入自动标记 `provenance=ai`。
5. **部署**：生产级 Docker 多阶段构建、完整 Docker Compose、Helm Chart、GitHub Actions CI/CD、生产环境配置管理。

### 1.3 不在本版本范围

- 扫描件 OCR、音视频转写、图片理解
- Slack / 飞书 / Confluence / Notion 等外部连接器
- AI 策略学习、高级信任衰减模型
- 多租户 SaaS 级隔离（保留单实例 + 空间隔离）

## 3. 核心用户故事

按 5 个方向组织，格式：As a [role], I want [feature] so that [benefit]。

### 3.1 真实嵌入/解析

- **US-E1**：作为一名**知识库管理员**，我希望上传 PDF、DOCX、Markdown、TXT 后系统能自动提取正文、标题、作者等元数据并合理分块，这样我能一次性导入现有文档资产而不必手动复制粘贴。
- **US-E2**：作为一名**知识工作者**，我希望搜索时系统使用真实嵌入模型理解语义，而不是仅匹配关键词，这样我能找到“意思相近但措辞不同”的知识。
- **US-E3**：作为一名**运维工程师**，我希望嵌入模型可配置（本地 ONNX / 云端 API / 多语言模型），这样我能根据硬件、合规和语种需求选择合适方案。

### 3.2 协同编辑

- **US-C1**：作为一名**文档编辑者**，我希望与同事同时编辑同一条记忆/文档时能看到对方光标和选区，这样我们能像在线文档一样协同写作。
- **US-C2**：作为一名**团队负责人**，我希望每次保存或重大修改都留下版本历史，并能对比差异、回退到旧版本，这样我能追溯内容变更并防止误操作。
- **US-C3**：作为一名**知识工作者**，我希望当 concurrent edits 产生冲突时系统能高亮冲突区域并提供“接受我的/接受对方的/合并”三种操作，这样我能主动解决分歧而不是被静默覆盖。
- **US-C4**：作为一名**编辑者**，我希望 AI 在协同编辑时实时召回相关知识并在侧边栏提示潜在矛盾，这样我能保持新内容与既有结论一致。

### 3.3 权限

- **US-P1**：作为一名**IT 管理员**，我希望用户通过 JWT/SSO 登录后系统根据全局角色（Admin/Owner/Editor/Viewer）决定可访问的功能，这样我能统一管控企业账号。
- **US-P2**：作为一名**空间所有者**，我希望邀请成员并分配空间角色（Owner/Editor/Viewer），设置空间可见性为公开/团队/私有，这样我能按项目或部门隔离知识。
- **US-P3**：作为一名**敏感文档作者**，我希望对单条记忆设置更细粒度的可见范围（继承空间/仅自己/指定成员），这样我能控制核心结论不被误传。
- **US-P4**：作为一名**合规人员**，我希望所有写操作留下审计日志（谁、何时、做了什么），这样我能满足企业审计要求。

### 3.4 Agent 集成

- **US-A1**：作为一名**Agent 开发者**，我希望通过 API Key 调用稳定的 REST API 搜索和写入记忆，这样我的 Agent 能复用组织知识库。
- **US-A2**：作为一名**Agent 开发者**，我希望 Agent 写入的记忆自动标记 `provenance=ai` 并进入 pending 审核，这样人类能清楚区分 AI 贡献并把关质量。
- **US-A3**：作为一名**Agent 开发者**，我希望 epicode-kb 提供 MCP（Model Context Protocol）Server，让我的 LLM 应用通过标准协议读取/写入记忆，这样我就不必维护自定义集成。

### 3.5 部署

- **US-D1**：作为一名**运维工程师**，我希望通过一条 `docker compose up` 在本地或单节点服务器启动包含后端、前端、数据库、向量扩展的完整环境，这样我能快速私有化部署。
- **US-D2**：作为一名**SRE**，我希望项目提供 Helm Chart，能在 Kubernetes 上完成安装、滚动更新、扩缩容和配置/密钥管理，这样我能接入现有 K8s 交付流程。
- **US-D3**：作为一名**发布经理**，我希望 GitHub Actions 在 push/PR 时自动完成构建、测试、镜像推送、Helm 打包和 Release 发布，这样我能实现持续交付。

## 4. 需求池

优先级定义：

- **P0**：v0.2.0 发布必须有，阻塞上线。
- **P1**：重要，建议本版本内完成；若资源不足可放到 v0.2.1。
- **P2**：锦上添花，后续版本再考虑。

### 4.1 P0 — 必须有

| ID | 方向 | 需求 | 验收标准 |
|----|------|------|----------|
| EMB-01 | 真实嵌入 | 接入 ONNX 真实嵌入模型，替换 `RandomEmbedder` | `cargo test` 通过；同文本两次嵌入余弦相似度 ≈ 1.0；语义搜索 top-1 命中同主题文档 |
| EMB-02 | 文档解析 | 支持 Markdown、TXT、PDF、DOCX 入库，提取标题/作者/页码/章节等元数据 | 四种格式上传均成功；PDF 不再依赖 ASCII fallback；DOCX 保留标题层级 |
| EMB-03 | 分块 | 可配置 chunk size、overlap、按句子边界切分；chunk 携带 source/heading/page 元数据 | chunk 平均长度可控；元数据在记忆详情中可见 |
| COL-01 | 协同编辑 | 基于 WebSocket + CRDT 的实时协同编辑器；支持多人同时编辑同一记忆 | 两个客户端同时编辑，双方内容在 <300ms 内同步；不出现丢失 |
| COL-02 | 版本历史 | 每次保存生成版本快照；支持 diff 对比与一键回退 | 版本列表可查看；回退后内容与旧版本一致 |
| COL-03 | 冲突解决 | 并发保存冲突时展示冲突区域，提供 accept-mine/accept-theirs/merge 操作 | 冲突 UI 出现；选择后生成 `provenance=co` 的新版本 |
| PER-01 | 认证 | JWT 登录/登出；Token 刷新；支持用户名密码或接入 OIDC（至少一种） | 未登录访问受保护接口返回 401；登录后获得有效 JWT |
| PER-02 | RBAC | 全局角色 + 空间角色；中间件对所有写接口执行权限校验 | Viewer 无法调用 POST/PUT/DELETE；越权返回 403 |
| PER-03 | 可见性 | 空间支持 public/team/private；记忆支持 inherit/space-only/private/selected | 私有空间内容对其他空间成员不可见；私有记忆对无权限者隐藏 |
| PER-04 | 权限管理 UI | 管理员/空间所有者可在前端管理成员、角色、可见性 | 成员邀请、角色变更、空间可见性切换可用 |
| AGT-01 | Agent API | 独立的 Agent API Key 认证；提供搜索记忆、写入记忆、获取单条记忆接口 | API Key 可创建/撤销；Agent 写入记忆 `provenance=ai` |
| AGT-02 | MCP 接口 | 实现 MCP Server，暴露 `search_memories`、`remember`、`get_memory` 工具 | MCP Client 可成功调用；返回符合协议规范 |
| DEP-01 | Docker 构建 | 后端/前端多阶段 Dockerfile，镜像非 root 运行，带 healthcheck | `docker build` 通过；镜像可启动 |
| DEP-02 | Docker Compose | 编排后端、前端、PostgreSQL + pgvector、可选 Nginx；支持 secrets/env 管理 | `docker compose up` 一键启动；数据持久化到 volume |
| DEP-03 | Helm Chart | 提供 K8s Deployment/Service/Ingress/ConfigMap/Secret/HPA/Migration Job | `helm install` 成功；滚动更新无中断 |
| DEP-04 | CI/CD | GitHub Actions：fmt/clippy/test、前端 typecheck/build、Docker buildx 多架构推送、Helm package、Release | PR 触发检查；tag push 自动发布镜像和 Helm chart |
| DEP-05 | 生产配置 | 支持 dev/staging/prod 配置分离；敏感信息通过环境变量或 K8s Secret 注入 | 同一镜像在不同环境通过配置启动；日志中无密钥泄露 |

## 5. UI/UX 设计要点

v0.2.0 需要新增/改造以下页面与组件：

| 页面/组件 | 说明 | 优先级 |
|-----------|------|--------|
| **登录页** | JWT 表单或 OIDC 登录入口；登录后跳转原目标页 | P0 |
| **Admin 控制台** | 用户列表、全局角色分配、API Key 管理、审计日志入口 | P0 |
| **空间设置页** | 空间信息、可见性开关、成员管理、角色分配、AI 写入开关、保留策略 | P0 |
| **记忆详情/协同编辑页** | 富文本/CRDT 编辑器、在线用户头像/光标、版本历史抽屉、diff 与回退按钮、权限设置 | P0 |
| **冲突解决弹窗** | 并排对比冲突版本，提供 accept-mine / accept-theirs / edit-merge | P0 |
| **Agent 集成页** | 生成/撤销 API Key、MCP Server 地址与工具说明、Python/TS SDK 代码片段 | P0 |
| **导航与操作权限化** | 根据当前用户角色动态隐藏/禁用“编辑/删除/邀请/设置”按钮 | P0 |
| **审核队列页**（v1 占位升级） | 展示 AI/Agent 提交的待审核记忆，支持采纳/拒绝/修改 | P1 |
| **知识图谱页**（v1 占位升级） | Cytoscape.js 展示记忆关系与冲突边，按 provenance 着色 | P1 |

**关键交互原则**：

- 任何 AI/Agent 生成内容默认显示 `provenance=ai` 标签 + pending 状态，人类采纳后才提升 trust。
- 无权限的操作按钮不展示或禁用，避免用户误点后收到 403。
- 协同编辑时保存即生成版本，版本历史按时间线展示，diff 高亮增删改。

## 6. 非功能需求

### 6.1 性能

| 指标 | 目标 | 备注 |
|------|------|------|
| 单条嵌入延迟（P95） | < 500 ms | CPU 运行 all-MiniLM-L6-v2；可配置 GPU |
| 批量嵌入吞吐 | > 20 chunks/s | 异步队列场景下 |
| 混合检索延迟（P95，10 万记忆） | < 200 ms | 包含语义 + 全文 + 过滤 |
| 协同编辑同步延迟 | < 300 ms | 同一文档内两位用户 |
| 10 MB 文档解析入库 | < 30 s | 含解析、分块、嵌入、索引 |
| 并发用户支持 | 200+ | 单实例；生产可通过 K8s 横向扩展 |

### 6.2 安全

- **认证**：JWT access/refresh token，支持过期与吊销；API Key 按空间/能力范围授权。
- **授权**：RBAC 中间件覆盖所有 API；前端按钮按权限渲染。
- **传输与存储**：生产环境强制 HTTPS；密钥、API Key、LLM key 不存入代码仓库，通过 env/Secret 注入。
- **输入校验**：文件类型白名单、大小限制、文件名净化；SQL 注入由 SQLx/rusqlite 参数化查询防护。
- **审计**：记录所有写操作（创建/更新/删除/采纳/拒绝/权限变更）到 `audit_logs`。
- **Agent 安全**：Agent 写入默认 `review_status=pending`，不直接进入 accepted 搜索结果。

### 6.3 可观测性

- **日志**：结构化 JSON 日志，包含 request_id、user_id、space_id；错误日志带堆栈。
- **指标**：Prometheus `/metrics` 暴露 QPS、延迟、嵌入队列深度、搜索命中率、审核队列长度。
- **健康检查**：`/health/live`（存活）、`/health/ready`（依赖就绪，含 DB、Tantivy、模型加载）。
- **链路追踪**：OpenTelemetry trace，覆盖请求 → 解析 → 嵌入 → DB → Tantivy。
- **告警**：错误率 > 1%、P95 延迟 > 阈值、队列堆积、磁盘空间不足时触发告警。

## 7. 数据模型与接口变更要点

### 7.1 新增/变更表

| 表/字段 | 变更 | 说明 |
|---------|------|------|
| `memories` | 新增 `visibility` | `inherit` / `space_only` / `private` / `selected` |
| `memories` | 新增 `version_of` / `version_seq` | 版本历史链路 |
| `memory_versions` | 新表 | 版本快照、编辑者、diff |
| `spaces` | 新增 `visibility` | `public` / `team` / `private` |
| `spaces` | 新增 `slug`、`owner_id` | 空间管理与路由 |
| `space_members` | 保留，强化 role 校验 | Owner/Editor/Viewer |
| `memory_permissions` | 新表 | 单条记忆 ACL（user_id / role / permission） |
| `users` | 新增 `password_hash` / `sso_subject` / `is_active` | 支持本地与 OIDC |
| `api_keys` | 新表 | Agent 调用凭证（key_hash、scope、space_id、expires_at） |
| `audit_logs` | 已存在，需补全所有写操作记录 | 保留 |

### 7.2 关键 API 新增

| 方法 | 路径 | 说明 |
|------|------|------|
| `POST` | `/api/v1/auth/login` | JWT 登录 |
| `POST` | `/api/v1/auth/refresh` | Token 刷新 |
| `GET/POST` | `/api/v1/spaces/{id}/members` | 空间成员管理 |
| `PUT` | `/api/v1/spaces/{id}/visibility` | 空间可见性 |
| `PUT` | `/api/v1/memories/{id}/visibility` | 记忆可见性 |
| `GET` | `/api/v1/memories/{id}/versions` | 版本历史 |
| `POST` | `/api/v1/memories/{id}/versions/{vid}/revert` | 回退版本 |
| `WS` | `/api/v1/collab/{memory_id}` | 协同编辑 WebSocket |
| `POST` | `/api/v1/agents/search` | Agent 搜索接口（API Key） |
| `POST` | `/api/v1/agents/remember` | Agent 写入接口（API Key，强制 `provenance=ai`） |
| `GET` | `/api/v1/mcp/sse` | MCP Server SSE 端点 |

## 8. 待确认问题

| # | 问题 | 影响范围 | 建议方向 |
|---|------|----------|----------|
| Q1 | **嵌入模型选型**：默认继续用 `all-MiniLM-L6-v2`（英文 384 维），还是换用多语言/BGE 模型？中文企业场景是否需要单独选型？ | 解析/搜索质量、镜像大小、推理性能 | 默认保留 MiniLM 以保证兼容性；通过配置支持切换，v0.2.1 评估中文模型 |
| Q2 | **向量检索规模**：当前 SQLite 全量加载 embedding 计算。v0.2.0 是否必须引入 pgvector 或独立向量库（Qdrant/Milvus）？ | 架构、部署复杂度、性能 | Docker Compose 默认带 PostgreSQL+pgvector；SQLite 保留为开发/轻量模式 |
| Q3 | **协同编辑后端实现**：后端使用 Rust `yrs` + 自定义 Axum WebSocket，还是增加一个 Node y-websocket sidecar？ | 技术栈一致性 vs 开发成本 | 优先 `yrs` + Axum WS 保持纯 Rust 后端；若工期紧张改为 sidecar |
| Q4 | **MCP 传输方式**：MCP Server 采用 stdio（本地 Agent）还是 SSE/HTTP（远程 Agent）作为默认？ | Agent 接入形态、认证 | 先实现 stdio 与 SSE 两种；文档给出示例 |
| Q5 | **单条记忆 ACL 是否必须**：三级权限中的“记忆级”是否会在本版本中高频使用，还是先以空间级权限满足 80% 场景？ | 复杂度、性能、UX | P0 保留字段与基础校验；UI 与完整 ACL 放 P1 |
| Q6 | **OIDC 优先级**：SSO/OIDC 是企业刚需，但实现工作量较大，是否纳入 P0 还是 P1？ | 权限模块范围 | P0 完成 JWT + 本地账号；P1 接入 OIDC |

## 10. 主要风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| `ort`/`pdf-extract` 等 crate 编译困难或版本冲突 | 阻塞真实嵌入/解析 | 先以 feature flag 隔离；CI 多环境验证；必要时换用替代库 |
| 协同编辑 CRDT 实现复杂、延迟不达标 | 用户体验差 | 先实现基础 OT/CRDT，复杂场景保留“锁定编辑”兜底 |
| RBAC 引入后大量接口需改造，回归成本高 | 延期或 Bug | 先写权限中间件 + 统一注解/包装器；逐接口迁移并补测试 |
| MCP 协议快速演进 | 兼容性风险 | 紧跟 2024-12 MCP spec；抽象接口层便于后续升级 |
| 生产部署依赖 PostgreSQL/pgvector，增加运维门槛 | 小企业采用阻力 | Docker Compose 提供一键启动；文档包含备份/恢复脚本 |
