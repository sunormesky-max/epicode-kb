## 1. 版本目标与范围

### 1.1 一句话产品目标

> **将 epicode-kb 从"已完成基础设施的人机协同知识库"推进为具备 AI 主动整理、矛盾检测、健康度自感知、协同编辑增强五大高阶认知能力的完整闭环系统。**

### 1.2 v0.2.0 状态回顾

| 能力域 | v0.2.0 现状 | s1-s5 对应 |
|--------|-----------|-----------|
| 记忆溯源（provenance/trust/review） | ✅ 完整实现 | Phase 1 完成 |
| 权限（JWT+RBAC 三级） | ✅ 完整实现 | 基础设施 |
| 协同编辑（Tiptap+yrs） | ✅ 基础版 | Phase 5 基础 |
| 冲突解决（save_version/revert/resolve_conflict） | ✅ 基础版 | Phase 3 部分 |
| AI 提议系统（proposal 状态机+审核队列） | ⚠️ 骨架文件存在（`dream/proposal.rs`、`api/proposal.rs`），功能未接通 | Phase 2 待实现 |
| 矛盾检测（语义对比+冲突中心） | ❌ 未实现 | Phase 3 补全 |
| 知识健康度（衰减/缺口/孤岛/仪表盘） | ❌ 未实现 | Phase 4 待实现 |
| 编辑器上下文召回+实时矛盾+AI 侧栏 | ❌ 未实现 | Phase 5 补全 |
| 通知系统（webhook） | ⚠️ `notify/webhook.rs` 骨架存在，功能未接通 | Phase 4 依赖 |

### 1.3 v0.3.0 范围

按 priority 顺序推进 4 个 Phase 的增量能力：

| Phase | 名称 | 内容 | 新代码占比 |
|-------|------|------|-----------|
| **P2** | AI 提议系统 | Proposal 状态机 / dream cycle 改提议模式 / 审核队列 UI / 批量审核 / 反馈学习 | 30% |
| **P3** | 矛盾检测补全 | 语义矛盾检测算法 / Conflict 记忆 / 冲突中心 UI / 知识图谱矛盾边 | 25% |
| **P4** | 知识健康度 | 查询日志 / 衰减模型 / 缺口检测 / 孤岛检测 / 健康仪表盘 / 通知系统 | 25% |
| **P5** | 协同编辑增强 | 编辑器上下文召回 / 实时矛盾检测 / AI 侧边栏建议面板 | 20% |

### 1.4 不在本版本范围

- OIDC/SSO 集成（延至 v0.4.0）
- 外部连接器（Slack/飞书/Confluence/Notion）
- 多租户 SaaS 级隔离
- 音视频/图片理解
- 知识图谱 3D 可视化（延至 v0.4.0）

## 4. 关键交互流程

### 4.1 AI 提议审核流程

```
Dream Cycle 触发（定时/手动）
      ↓
扫描记忆空间 → 发现 merge 候选 / link 机会 / 过期内容 / 主题聚类
      ↓
生成 ai_proposals 记录（status=pending）
      ↓
维护者打开审核队列 → 看到提议卡片（类型+置信度+涉及记忆）
      ↓                     ↓                    ↓
   approve           reject（附理由）       modify（修改后采纳）
      ↓                     ↓                    ↓
执行合并/链接         记录反馈→调整频率     执行修改版操作
      ↓
生成 Co provenance 记忆（provenance_meta 含 proposal_id）
```

### 4.2 矛盾检测与解决流程

```
编辑者保存记忆 / dream cycle 扫描
      ↓
矛盾检测器启动：提取新内容→embedding→搜索近邻→语义距离<0.3 的候选
      ↓
LLM 事实对比：提取双方的关键断言→判断是否矛盾
      ↓
发现矛盾 → 创建 Conflict 记忆（provenance=conflict, conflicting_ids=[A, B]）
      ↓
维护者在冲突中心看到并排对比
      ↓
accept-A / accept-B / both-true → 解决记录作为 Co 记忆
```

### 4.3 知识健康监控流程

```
定时任务（每天）触发健康扫描
      ↓
├─ 统计：total_memories, human/ai/co 占比, avg_trust
├─ 衰减检测：last_accessed_at > 90天 且 access_count < 5 → stale
├─ 缺口检测：query_logs 中 result_count=0 的 top-N 查询 → gaps
├─ 孤岛检测：零入链 + 零出链 → orphan
└─ 综合评分：(活跃度×0.3 + 完整度×0.3 + 时效性×0.2 + 信任度×0.2)
      ↓
写入 knowledge_health 快照
      ↓
推送给空间维护者（Webhook / 通知中心）
```

## 5. 非功能需求

| 维度 | 要求 |
|------|------|
| **性能** | 矛盾检测单条 <500ms；全空间扫描 <30s（10K 记忆内）；健康扫描 <10s |
| **安全** | AI 提议禁止直接写入 memory，必须走 proposal→review 路径；审核操作记录 audit_log |
| **可观测性** | 新增 metrics：proposal_count、conflict_count、health_score、query_zero_result_rate |
| **兼容性** | 不影响 v0.2.0 已有功能；新增表通过 004_v3_schema.sql migration 幂等创建 |
| **LLM 依赖** | 事实对比调用 DeepSeek/Ollama；支持配置 `EPICODE_KB_LLM_PROVIDER` |
| **通知** | Webhook 签名验证（HMAC-SHA256）；邮件 TLS（P1） |

## 7. 待确认问题

| # | 问题 | 影响范围 | 建议 |
|---|------|---------|------|
| Q1 | **LLM 提供商**：矛盾检测的事实对比需要 LLM 调用来判断"两条内容是否矛盾"。当前项目已有 `llm/deepseek.rs` 和 `llm/ollama.rs`，默认使用哪个？ | 检测准确率、成本、隐私 | P0 默认 Ollama（本地免费）；支持 `EPICODE_KB_LLM_PROVIDER` 切换 DeepSeek |
| Q2 | **dream cycle 扫描频率**：当前是定时还是手动触发？ | P2 提议生成时机 | P0 默认手动触发 + 可配置定时（每天/每周）；避免自动模式产生过多噪声 |
| Q3 | **健康度评分公式**：活跃度/完整度/时效性/信任度的权重是否合理？ | 仪表盘 UX | 先按 30/30/20/20 实现；暴露配置权重参数，后续按用户反馈调整 |
| Q4 | **矛盾检测阈值**：语义距离<0.3 作为矛盾候选的阈值是否合适？ | 误报/漏报率 | P0 默认 0.3，配置项 `EPICODE_KB_CONFLICT_THRESHOLD`；后续用标注数据调优 |
| Q5 | **编辑器上下文召回频率**：实时 streaming 对后端压力大，如何平衡延迟与性能？ | P5 用户体验 | P1 阶段先用 debounce 3s + top-5 召回 + 缓存；避免每次 keystroke 都请求 |

## 8. 主要风险与缓解

| 风险 | 影响 | 缓解 |
|------|------|------|
| LLM 事实对比幻觉（误判矛盾） | 信任度下降 | 对比结果标注 `ai_detected`；人工确认后才创建 Conflict 记忆 |
| dream cycle 提议过多淹没审核队列 | UX 差 | 按置信度排序；每日上限；同类型提议聚合 |
| 健康度扫描性能退化 | 10K+ 记忆时超时 | 增量扫描（仅扫最近更新的记忆）；异步执行 |
| 通知系统爆炸（大量告警） | 用户疲劳 | 合并通知频率（每天汇总一次）；按严重度分级 |
| P5 上下文召回延迟高 | 编辑体验差 | debounce + 缓存最近结果 + 前端骨架屏 |
